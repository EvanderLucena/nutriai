# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Key principle:** The UI already exists (PlansView, FoodsView, all plan components). This contract **preserves** the existing visual language and specifies only the **new interactions** needed for API-wired version (auto-save feedback, loading/error states, portion-picker, grams input, save-status toast).

## New Interaction Specifications

These are the interactions that **don't exist in the current prototype** and must be added for API-wired version.

### 5.1 Auto-Save Feedback (replaces manual Save button)

**Current state:** PlansView has a manual "Salvar" button + dirty flag (`NÃO SALVO` / `SALVO`).

**New behavior (D-18):** Each edit action (add food, change qty, remove meal, edit option name, add/remove option, edit extras) triggers an immediate API call. No bulk Save button.

| Element | Specification |
|---------|---------------|
| Save-status indicator | Replace `NÃO SALVO` / `SALVO` text with 3-state indicator positioned top-right of plan header |
| States | `SALVO` (fg-subtle) → `SALVANDO…` (fg-muted, appears on mutation start) → `SALVO` (fg-subtle, appears on success) or `ERRO AO SALVAR` (coral) |
| Toast on error | Small toast at bottom-right: `Erro ao salvar — tente novamente`. Auto-dismisses after 4s. Background: `var(--surface)` with `1px solid var(--coral)` border. |
| Optimistic updates | UI updates immediately on user action; API call happens in background. On error, revert to previous state and show error toast. |
| MealFood inline edit | On blur of any inline field (food, qty, prep, kcal, etc.), trigger auto-save for that item via `PATCH /api/v1/patients/{id}/plan/meals/{mealId}/options/{optionId}/items/{itemId}` |
| Plan title edit | On blur or Enter, trigger `PATCH /api/v1/patients/{id}/plan` with updated title |
| Macro targets edit | On blur of any daily macro field, trigger plan-level PATCH |

### 5.2 Portion Picker in AddFoodModal

**Current state:** AddFoodModal lets you search the catalog, select a food, and type a free-text quantity. Macros are copied directly from per100 or preset nutrition.

**New behavior (D-05, D-06, D-07):** When a base food is selected, show portion references as clickable chips. Clicking a portion auto-fills qty text and grams number. Nutritionist can also type grams directly. All calculation goes through grams → per100.

| Element | Specification |
|---------|---------------|
| Food search results | Same as current (list with highlight), but results come from API instead of `FOODS_CATALOG` |
| Portion chips | When a base food is selected with portions, render chips below the qty input. Each chip: pill shape, `1px solid var(--border)`, `border-radius: 999`, `font-size: 11px`, `padding: 3px 7px`. Text: `{portion.name} ({portion.grams}g)` |
| Portion chip click | Fills qty input with portion name (e.g. "1 unidade") AND fills grams input with portion.grams value |
| Grams input | New numeric input below qty. Label: `Gramas` (eyebrow). Placeholder: `0`. Type: number. When portions is selected, auto-filled. Nutritionist can override. |
| Qty input | Free-text human-readable portion description. Placeholder: `ex: 1 unidade, 2 col. sopa`. Not used for calculation — display only. |
| Macro preview | After food + grams are set, show the calculated macros in the preview grid (but note: backend calculates final values — frontend preview is approximate from per100 × grams / 100) |
| Add button | Sends `foodId` + `grams` + `qty` to API. Backend calculates and returns frozen macros. UI updates with server response. |
| Preset food selection | When a preset is selected, grams is pre-filled from `food.grams`, qty from `food.portionLabel`. No portion chips shown. |

### 5.3 PlanFoodRow inline editing (Grams integration)

**Current state:** PlanFoodRow shows inline inputs for food name, qty, prep, kcal, prot, carb, fat. All fields directly editable.

**New behavior:** Food name becomes read-only (frozen from catalog). Grams field added as numeric input for calculation. Macro fields (kcal, prot, carb, fat) become read-only (backend-calculated, frozen). Qty remains free-text editable.

| Column | Editability | Behavior on change |
|--------|-------------|-------------------|
| Alimento (food) | **Read-only** | Display `foodName` (frozen at add time). Clicking shows tooltip: food is linked to catalog entry. |
| Quantidade (qty) | Editable (free text) | On blur → auto-save qty field only (display text, not calculation) |
| ✦ Gramas (grams) | **New column, editable** (numeric) | On blur → triggers API recalculation. Backend returns updated frozen macros. UI updates with server response. |
| Preparo (prep) | Editable (free text) | On blur → auto-save prep field only |
| Kcal / Prot / Carb / Gord | **Read-only** | Frozen macro values from backend. Not directly editable. Gray background to indicate read-only. |
| Remove button | Clickable | Confirms via DeleteConfirmModal, then removes |

**Table column layout change:**
- Current: `2.2fr 1fr 1.8fr 0.8fr 0.8fr 0.9fr 0.8fr 28px` (8 cols)
- New: `2.2fr 1fr 0.8fr 1.8fr 0.8fr 0.8fr 0.9fr 0.8fr 28px` (9 cols) — inserted `0.8fr` grams column between qty and prep

### 5.4 Loading States

| Component | Loading State |
|-----------|---------------|
| PlansView (plan data) | Show skeleton: left sidebar skeleton (6 meal slots) + right content skeleton (option tabs + table header + 3 rows + totals bar). Use pulse animation on `var(--surface-2)` background blocks. |
| FoodsView (catalog data) | Show 6 skeleton cards in the `foods-grid`. Each skeleton: card shape with pulse animation. |
| AddFoodModal search | Show 4 skeleton rows in search results while debounced search runs (300ms debounce). |
| Inline macro recalc | After grams change, macros flash briefly with `opacity: 0.5` transition then resolve with new values. Duration: ~200ms. |

### 5.5 Plan Auto-Creation on Patient Creation

**New behavior (D-13, D-14):** When a new patient is created, the backend auto-creates a plan with 6-meal template. The nutritionist navigating to the Plans tab sees the pre-populated template.

| Element | Specification |
|---------|---------------|
| Template meals | Café da manhã (07:00), Lanche manhã (10:00), Almoço (12:30), Lanche tarde (15:30), Jantar (19:30), Ceia (22:00) |
| Template options | Each meal starts with one empty option: `Opção 1 · Clássico` (no food items) |
| Macro targets | Distributed proportionally. Default total: varies by patient objective (backend-calculated). All meals start at 0/0/0/0 actual vs target. |
| Observations | Empty field, editable |

### 5.6 Extras Section Changes (D-11, D-12)

**Current state:** ExtrasSection has columns: Item, Porção, Categoria, Kcal, Prot, Carb, Gord, ✕. Each column is inline editable. Extras have `category` and `note` fields.

**New behavior:** Remove `category` and `note` columns. Extras are simplified: name + qty + macros only.

| Column | Editability | Notes |
|--------|-------------|-------|
| Item (name) | Editable | Free text |
| Porção (qty) | Editable | Free text, e.g. "350ml · lata" |
| Kcal | Editable (numeric) | Auto-save on blur |
| Prot | Editable (numeric) | Auto-save on blur |
| Carb | Editable (numeric) | Auto-save on blur |
| Gord | Editable (numeric) | Auto-save on blur |
| ✕ | Clickable | Remove (no confirmation for extras — lightweight action) |

**Column layout change:**
- Current: `1.5fr 1fr 1fr 0.7fr 0.8fr 0.9fr 0.8fr 28px` (8 cols with category)
- New: `1.5fr 1fr 0.7fr 0.8fr 0.9fr 0.8fr 28px` (7 cols — category and note removed)

### 5.7 FoodsView API Wiring

| Change | Specification |
|--------|---------------|
| Data source | Replace `FOODS_CATALOG` import with `useFoodCatalog()` TanStack Query hook |
| Search | Debounced 300ms, server-side via `?search=` parameter |
| Category filter | Server-side via `?category=` parameter |
| Pagination | Server-side via `?page=&size=12` (existing `FOODS_PAGE_SIZE=12`) |
| Create food | `POST /api/v1/foods` → invalidate catalog query |
| Edit food | `PATCH /api/v1/foods/{id}` → invalidate catalog query |
| Delete food | `DELETE /api/v1/foods/{id}` via `DeleteConfirmModal` → invalidate catalog query |
| "usado em N planos" | `food.used` count returned from API (denormalized or computed). Display preserved. |

### 5.8 Plan-Level Interactions

| Interaction | Specification |
|-------------|---------------|
| Edit plan title | Click title → inline input → blur/Enter → `PATCH /api/v1/patients/{id}/plan` with `{ title }` |
| Edit daily macro targets | Click any DailyMacro value → inline input → blur/Enter → `PATCH /api/v1/patients/{id}/plan` with `{ kcalTarget, protTarget, carbTarget, fatTarget }` |
| Edit observations | Click observations text → inline edit → blur → `PATCH` plan with `{ notes }` |
| Add meal slot | AddMealModal → `POST /api/v1/patients/{id}/plan/meals` → invalidate plan query |
| Remove meal slot | DeleteConfirmModal → `DELETE /api/v1/patients/{id}/plan/meals/{mealId}` → invalidate |
| Add option | Click "Nova opção" → `POST .../options` with `{ name: "Opção N · Cópia" }` → invalidate |
| Remove option | OptionTab ✕ button (if >1 option) → `DELETE .../options/{optionId}` → invalidate |
| Rename option | Double-click OptionTab → inline edit → blur → `PATCH .../options/{optionId}` with `{ name }` |
| Add food to option | AddFoodModal → `POST .../items` with `{ foodId, grams, qty }` → backend returns frozen macros → invalidate |
| Remove food from option | DeleteConfirmModal → `DELETE .../items/{itemId}` → invalidate |
| Add extra | "Adicionar Opção Extra" → `POST .../extras` with blank extra → invalidate |
| Remove extra | ✕ button → `DELETE .../extras/{extraId}` → invalidate (no confirm for extras) |
| Edit extra fields | On blur of any inline field → `PATCH .../extras/{extraId}` with changed field → auto-save |
