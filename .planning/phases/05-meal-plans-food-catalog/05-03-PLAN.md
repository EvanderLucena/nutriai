# Refatoração: Unificação do modelo Food + MealFood

## Motivação

O modelo original tinha BASE (por 100g) e PRESET (porção pronta) como tipos separados, com 10+ campos condicionais e lógica de branch em todo o codebase. O nutricionista na prática cadastra sempre uma "referência" do alimento com seus macros, e depois ajusta a quantidade no plano. Simplificar para um único modelo elimina complexidade desnecessária.

## Decisões de Design

1. **Food unificado**: um só tipo de alimento. O nutri cadastra com macros da porção de referência.
2. **Sistema de unidades**: GRAMAS, UNIDADE, ML — o cálculo é unit-agnostic (`macro * amount / referenceAmount`).
3. **Cálculo proporcional**: sempre `macro * quantidadeNoPlano / referenceAmount`. Não importa se é grama, ml ou unidade.
4. **prep no Food**: preparo sugerido que é copiado para o MealFood ao adicionar no plano (editável lá).
5. **Portions removido**: o atalho de porções é substituído pelo `portionLabel` + `unit` + `referenceAmount`.
6. **MealFood.grams → amount + unit**: campo `grams` vira `amount` (numérico) + `unit` (GRAMAS/UNIDADE/ML).

### Cálculo no plano

```
kcalNoPlano = food.kcal * mealFood.amount / food.referenceAmount
```

Exemplos:
- Frango (ref=100g, kcal=165): plano=80g → 165 * 80/100 = 132
- Omelete (ref=1 unid, kcal=358): plano=1.5 unid → 358 * 1.5/1 = 537
- Leite (ref=250ml, kcal=150): plano=200ml → 150 * 200/250 = 120

## Segurança / LGPD

- Sem dado sensível nos campos novos (unit, referenceAmount são dados nutricionais, não PII)
- Migration não altera dados de paciente
- `prep` é texto livre — sanitizar no service (trim, max length) igual aos outros campos String
- `unit` é enum fechado (validação no service) — sem risk de injection

## Riscos

- **Dados em produção**: migration V10 precisa de UPDATE para migrar dados existentes. Como estamos em dev, tabelas são recriadas. Se houver dados reais, adicionar UPDATE statements (documentados acima).
- **MealFood existentes**: UPDATE `meal_food SET amount = grams` na migration garante que dados existentes não se perdem.
- **Breaking change na API**: frontend e backend precisam ser deployados juntos ou a API quebra.
