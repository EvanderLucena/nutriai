    **backend/Dockerfile** — Multi-stage production build:
    - Stage 1 (build): eclipse-temurin:21-jdk-alpine, copy build.gradle + source, run `./gradlew bootJar`
    - Stage 2 (run): eclipse-temurin:21-jre-alpine, copy the JAR, expose 8080, `ENTRYPOINT ["java", "-jar", "app.jar"]`
    - This is the production Dockerfile. Phase 10 will use it for CI/CD.

    **backend/Dockerfile.dev** — Development image:
    - Base: eclipse-temurin:21-jdk-alpine
    - Install Gradle wrapper and dependencies (run `./gradlew dependencies` as a caching layer)
    - Expose 8080
    - Command: `./gradlew bootRun` with Spring DevTools for hot reload
    - Per D-20, Spring DevTools auto-restart watches for class file changes

    **frontend/Dockerfile.dev** — Development image:
    - Base: node:20-alpine
    - Copy package.json + package-lock.json first (cache layer), then `npm install`
    - Copy source files
    - Expose 5173
    - Command: `npm run dev -- --host 0.0.0.0` (host flag needed for Docker networking)
    - Per D-20, Vite HMR works via volume mount

    **docker/docker-compose.yml** — Main compose file per D-17 (3 services: frontend, backend, postgres):
    ```yaml
    services:
      postgres:
        image: postgres:17-alpine
        container_name: nutriai-postgres
        environment:
          POSTGRES_DB: ${POSTGRES_DB:-nutriai}
          POSTGRES_USER: ${POSTGRES_USER:-nutriai}
          POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-changeme}
        ports:
          - "${POSTGRES_PORT:-5432}:5432"
        volumes:
          - nutriai-pgdata:/var/lib/postgresql/data
        healthcheck:
          test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-nutriai}"]
          interval: 5s
          timeout: 5s
          retries: 5
        networks:
          - nutriai

      backend:
        build:
          context: ../backend
          dockerfile: Dockerfile.dev
        container_name: nutriai-backend
        environment:
          SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/${POSTGRES_DB:-nutriai}
          SPRING_DATASOURCE_USERNAME: ${POSTGRES_USER:-nutriai}
          SPRING_DATASOURCE_PASSWORD: ${POSTGRES_PASSWORD:-changeme}
          SERVER_PORT: 8080
        ports:
          - "8080:8080"
        depends_on:
          postgres:
            condition: service_healthy
        networks:
          - nutriai

      frontend:
        build:
          context: ../frontend
          dockerfile: Dockerfile.dev
        container_name: nutriai-frontend
        ports:
          - "5173:5173"
        depends_on:
          - backend
        networks:
          - nutriai

    volumes:
      nutriai-pgdata:

    networks:
      nutriai:
        driver: bridge
    ```

    **docker/docker-compose.dev.yml** — Development overrides per D-20 (hot reload):
    ```yaml
    services:
      backend:
        volumes:
          - ../backend/src:/app/src
        environment:
          SPRING_DEVTOOLS_RESTART_ENABLED: "true"
          SPRING_PROFILES_ACTIVE: dev

      frontend:
        volumes:
          - ../frontend/src:/app/src
          - ../frontend/public:/app/public
        environment:
          - VITE_API_URL=http://localhost:8080
    ```

    **Important Docker networking note:** In docker-compose, the backend connects to PostgreSQL using hostname `postgres` (the service name), NOT `localhost`. The `SPRING_DATASOURCE_URL` environment variable in docker-compose.yml uses `postgres:5432`. The `application.yml` default uses `localhost:5432` for running backend outside Docker.

    **scripts/seed-db.sh** — Per D-22, optional seed script that:
    - Connects to the PostgreSQL container via `docker compose exec postgres`
    - Runs a SQL file that inserts a test nutritionist (email: dev@nutriai.com, hashed password)
    - Only runs when invoked explicitly, NOT automatically on startup
    - Creates `/docker/seed-dev.sql` with the INSERT statement

    **Update scripts/dev.sh** (from Plan 01) — Modify to use `docker compose -f docker/docker-compose.yml -f docker/docker-compose.dev.yml up`. This ensures dev overrides (volume mounts, hot reload) are applied.

    **Update scripts/down.sh** — Modify to reference the correct compose files.

    **Update scripts/logs.sh** — Modify to use `docker compose -f docker/docker-compose.yml -f docker/docker-compose.dev.yml logs`.

    **Note on backend DevTools hot reload inside Docker:** Spring DevTools needs the source files to be mounted and the Gradle build to detect changes. The most practical approach for Docker is: mount the `src/` directory, and use `./gradlew bootRun` with DevTools enabled. Changes to Java files will trigger a recompile and restart. For faster feedback in dev, the developer can also run the backend outside Docker (just PostgreSQL in Docker) by setting SPRING_DATASOURCE_URL to `jdbc:postgresql://localhost:5432/nutriai` in application-dev.yml. Document this in a comment in docker-compose.dev.yml.
